- **`node` and `npm` not found in Claude's non-interactive shell (pre-Step 1)**

  After nvm was installed and `nvm alias default 20` was set, running
  `node --version` from Claude's Bash tool returned `command not found` even
  though the same command worked fine in a normal terminal.

  **Why this happens**

  nvm is not a traditional system package — it is a shell function, not a
  binary. When you install nvm, it adds a block of code to your `~/.bashrc`
  (or `~/.zshrc`) that looks like this:

  ```bash
  export NVM_DIR="$HOME/.nvm"
  [ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"
  ```

  This code runs every time you open a new interactive terminal, which is why
  `node` works in your shell immediately after setting the alias. However,
  Claude's Bash tool spawns a non-interactive, non-login shell. Bash has a
  rule: non-interactive shells do not source `~/.bashrc`. The nvm initialization
  code never runs, so `nvm` is never loaded, and the Node.js binaries it
  manages are never added to `PATH`.

  **The fix used during this session**

  Every `node` or `npm` command is prefixed with `source ~/.nvm/nvm.sh &&`:

  ```bash
  source ~/.nvm/nvm.sh && node --version
  source ~/.nvm/nvm.sh && npm install
  source ~/.nvm/nvm.sh && npm run dev
  ```

  Sourcing `nvm.sh` manually loads the nvm shell function and adds the active
  Node.js version's `bin/` directory to `PATH` for that command. It takes
  about 50ms and has no side effects.

  **If you are recreating these steps yourself in a terminal**

  You do not need the prefix. A normal interactive terminal sources `~/.bashrc`
  automatically when it opens, so nvm is already loaded and `node` / `npm` are
  on your PATH. Just run the commands as written in the implementation steps
  (without the `source ~/.nvm/nvm.sh &&` prefix).

  If you ever open a shell and `node` is missing unexpectedly (can happen in
  some CI environments or when SSHing without a login shell), running
  `source ~/.nvm/nvm.sh` once in that session fixes it.

  **Longer-term fix**

  If you want Claude's Bash tool to find `node` automatically without the
  prefix, add the nvm initialization lines to `~/.bash_profile` (in addition
  to `~/.bashrc`). Bash sources `~/.bash_profile` for login shells even when
  non-interactive. This is a system-level change outside the project and was
  not made here to avoid affecting other tools.

- **Vite 8 scaffolds a richer default `App.jsx` than expected (Step 1)**

  Running `npm create vite@latest` with the `react` template produces a
  fully styled demo page — logos, a click counter, links to documentation.
  This is useful to confirm the project works out of the box but is not
  related to Timer 2.0, so the entire file was replaced.

  The replacement `App.jsx` imports `Routes` and `Route` from React Router and
  renders a `Placeholder` component for every page the app will eventually
  have. This gives us a working route tree immediately so we can navigate to
  `/login`, `/surgeons`, `/operations/42/ocs2`, etc. in the browser and see a
  stub page rather than a blank screen or a 404.

  **Why define all routes now instead of one at a time?**

  React Router's `<Routes>` tree is defined centrally in `App.jsx`. If a URL
  has no matching `<Route>`, React Router renders nothing (a blank page). By
  stubbing every route upfront, any page in the app is at least visitable
  during development, which makes it easy to test navigation before the real
  page components are written.

- **`index.css` replaced (Step 1)**

  Vite's default `index.css` contained a full custom design system — CSS
  variables for colors, dark-mode media queries, typography settings, and
  layout rules for the demo page. All of it targeted the Vite demo and would
  have conflicted with Bootstrap. It was replaced with a single rule:
  `body { margin: 0; }`. Bootstrap handles everything else.

- **`node_modules/` and `frontend/dist/` added to `.gitignore` (Step 1)**

  `node_modules/` is the JavaScript equivalent of `.venv/` — it contains all
  installed packages and should never be committed. It can be several hundred
  megabytes and is fully reproducible from `package-lock.json` by running
  `npm install`.

  `frontend/dist/` is the production build output (the compiled, minified
  JavaScript and CSS). It is generated by `npm run build` and served by nginx
  in Docker. Committing it would mean the repo tracks generated files that
  change on every build — the Docker multi-stage build regenerates it from
  source anyway, so there is no value in versioning it.

  Both entries were added to the root `.gitignore` (the one at
  `timer-2.0/.gitignore`) rather than a new `frontend/.gitignore`, since the
  root file is already tracked and covers the whole repo.

- **Login cookies made the "unauthenticated logout" test authenticate itself (Step 2)**

  `test_unauthenticated_logout_returns_401` was written in Milestone 3 when
  the only way to authenticate was an `Authorization: Bearer` header. The test
  logged in via `api_client`, captured the tokens, then immediately used that
  same `api_client` to attempt a logout without setting any `Authorization`
  header — expecting a 401.

  After Step 2, the login response sets `timer_access` and `timer_refresh` as
  cookies. DRF's `APIClient` (the test HTTP client) stores cookies on the
  client object between requests, exactly as a real browser would. So the
  second request — the supposedly unauthenticated logout — now automatically
  carried the `timer_access` cookie, was authenticated via
  `CookieJWTAuthentication`, and returned 204 instead of 401.

  **The fix:** call `api_client.cookies.clear()` before the unauthenticated
  request. This simulates a fresh browser session with no stored cookies —
  the intent of the test — without changing any behavior the test was actually
  designed to verify.

  **Why this matters:** this is a good example of a test that was correct for
  its context but became invalid when the auth mechanism changed. The failure
  wasn't a regression in the application — the application was doing exactly
  the right thing (accepting cookie-authenticated requests). The test just
  needed to be updated to match the new reality.

- **`CookieTokenRefreshView` does not subclass `TokenRefreshView` (Step 2)**

  The simplest path would have been to subclass `TokenRefreshView` and
  override its data source. However, `TokenRefreshView` reads exclusively from
  `request.data` via its serializer, and `request.data` in DRF is an immutable
  parsed representation of the request body — you cannot inject values into it.

  Instead, `CookieTokenRefreshView` is a plain `APIView` that instantiates
  `TokenRefreshSerializer` directly with `{'refresh': refresh_token}`, where
  `refresh_token` comes from the cookie or the body. This matches what the
  base class does internally and keeps the behavior identical while giving full
  control over where the token value comes from.

  The `TokenError → InvalidToken` re-raise pattern is also copied from the
  base class. `TokenError` is a plain Python exception; `InvalidToken` is a
  DRF `APIException` with a 401 status code that DRF's exception handler knows
  how to convert into a proper JSON error response.

- **`<Outlet />` vs `children` props in React Router v6 (Step 3)**

  There are two patterns for wrapping routes in React Router v6:

  **Pattern A — `children` prop (older / simpler):**
  ```jsx
  <Route path="/" element={<PrivateRoute><Dashboard /></PrivateRoute>} />
  ```
  `PrivateRoute` receives `<Dashboard />` as `props.children` and either renders
  it or returns `<Navigate to="/login" />`. Simple, but every route must be
  individually wrapped.

  **Pattern B — `<Outlet />` with nested routes (idiomatic v6):**
  ```jsx
  <Route element={<PrivateRoute />}>
    <Route path="/" element={<Dashboard />} />
    <Route path="/surgeons" element={<Surgeons />} />
  </Route>
  ```
  `PrivateRoute` renders `<Outlet />` instead of `props.children`. React Router
  replaces `<Outlet />` with whichever child route matched the current URL.
  Adding a new authenticated page is just adding a new `<Route>` inside the
  group — the auth guard applies automatically.

  Timer 2.0 uses Pattern B. The structure in `App.jsx` is:
  ```
  PrivateRoute (auth guard)
    └── Layout (navbar + <Outlet />)
          └── [matched page component]
  ```
  `PrivateRoute` and `Layout` each render `<Outlet />`, which React Router
  fills with the next level down. This composability is the main reason the
  pattern exists.

- **`isLoading` prevents a redirect flash on page load (Step 3)**

  Without the `isLoading` state, every page load would briefly show the login
  redirect before `/auth/me/` responds.

  On first render, `user` is `null` (no user yet). Without `isLoading`,
  `PrivateRoute` would immediately see `user === null` and redirect to `/login`
  — even when the user IS logged in and the `/me/` call is just in flight.
  The user would see a flash of the login page before being sent back to their
  destination.

  With `isLoading: true` on mount, `PrivateRoute` renders a "Loading…" message
  instead of making any routing decision until the `/me/` call resolves. Once
  it resolves (either with user data or a 401), `isLoading` is set to `false`
  and the correct route is rendered — no flash.

- **Why `AuthProvider` wraps the entire `<Routes>` tree (Step 3)**

  React's context system works by looking up the component tree. A component
  that calls `useAuth()` looks for the nearest `AuthContext.Provider` ancestor.
  If `AuthProvider` only wrapped the authenticated routes (inside
  `<PrivateRoute>`), the public `/login` page — which also needs to call
  `auth.login()` — would have no provider and `useAuth()` would return `null`.

  Wrapping the entire `<Routes>` tree (in `App.jsx`) means every component in
  the app can call `useAuth()`, regardless of whether its route is public or
  protected.

- **Controlled vs uncontrolled form inputs in React (Step 4)**

  React forms can be written in two ways:

  **Uncontrolled** — the DOM owns the value; you read it with a `ref` when
  you need it:
  ```jsx
  const inputRef = useRef()
  // later: inputRef.current.value
  <input ref={inputRef} type="text" />
  ```

  **Controlled** — React state owns the value; the input is just a reflection
  of that state:
  ```jsx
  const [username, setUsername] = useState('')
  <input value={username} onChange={e => setUsername(e.target.value)} />
  ```

  Timer 2.0 uses controlled inputs throughout. Controlled inputs are slightly
  more verbose but give you the current value at all times (no ref gymnastics),
  make validation easy (just check the state), and make it straightforward to
  reset the form (just set state back to `''`).

- **`e.preventDefault()` is required for form submission in React (Step 4)**

  Without `e.preventDefault()`, submitting an HTML form causes a full browser
  page reload — the browser's default behavior for `<form>` elements. The page
  reload clears all React state and cancels any in-flight requests.

  Calling `e.preventDefault()` in the `onSubmit` handler cancels that browser
  default and lets the JavaScript handler take over completely. Every form in
  the app follows this pattern.

- **Why `autoFocus` is on the username field (Step 4)**

  `autoFocus` tells the browser to place the cursor in that input when the
  page loads. On the login page, the first thing a user always needs to do is
  type their username, so moving focus there immediately saves one click. It is
  a small UX detail but significant in a clinical environment where speed
  matters.

- **`autoComplete` attributes improve browser credential management (Step 4)**

  `autoComplete="username"` and `autoComplete="current-password"` are
  standardized values that tell the browser (and password managers like
  1Password or Bitwarden) what kind of data these fields hold. With these
  attributes set correctly, the browser can offer to fill the form
  automatically and can prompt to save new credentials after a successful
  login.

- **`username` field required a backend serializer change and a `select_related` fix (Step 5)**

  The milestone plan called for showing the linked Django username in the
  surgeon table. The `SurgeonSerializer` only exposed `id`, `first_name`,
  `last_name`, `email`, and `full_name` — `user` and `username` were not
  included. Adding `username` to the API response required two backend changes:

  1. A `SerializerMethodField` on `SurgeonSerializer`:
     ```python
     def get_username(self, obj):
         return obj.user.username if obj.user else None
     ```
     A regular `CharField(source='user.username')` was considered but would
     raise an error when `user` is `None` rather than returning `null`.
     `SerializerMethodField` gives full control over the null case.

  2. `select_related('user')` on the `SurgeonViewSet` queryset. Without it,
     Django would execute a separate `SELECT` for each surgeon's linked user
     row (the N+1 problem). With a list of 20 surgeons, that would be 21
     queries instead of 1. `select_related` tells Django to `JOIN` the user
     table in the initial query so all usernames are fetched at once.

- **React Query's `invalidateQueries` vs manually updating the cache (Step 5)**

  After a successful create, edit, or delete, the surgeon list needs to
  reflect the change. There are two approaches:

  **Manual cache update** — in `onSuccess`, call
  `queryClient.setQueryData(['surgeons'], oldData => ...)` to splice the new
  or removed item directly into the cached array. No network request is made.
  Fast, but requires careful handling of the paginated response structure.

  **Cache invalidation** — call
  `queryClient.invalidateQueries({ queryKey: ['surgeons'] })`. React Query
  marks the cached data as stale and immediately re-fetches from the server.
  Slightly slower (one extra GET), but the data shown is guaranteed to match
  what the server has — including any server-computed fields like `full_name`
  or constraints that rejected part of the input.

  Timer 2.0 uses invalidation throughout. For a dataset this small (a surgery
  center might have 10–30 surgeons), the extra request is imperceptible and the
  simplicity is worth it.

- **Bootstrap modals without Bootstrap JS (Step 5)**

  Bootstrap's default modal behavior requires its JavaScript bundle
  (`bootstrap.js` or `bootstrap.bundle.js`) to handle show/hide animations,
  focus trapping, and backdrop clicks. Importing that bundle into a React app
  creates a conflict: Bootstrap JS manipulates the DOM directly, while React
  owns the DOM. The two can fight over element state.

  The solution is to control the modal's visibility with React state and apply
  Bootstrap's CSS classes manually:
  ```jsx
  {showModal && (
    <>
      <div className="modal show d-block" tabIndex="-1">
        ...
      </div>
      <div className="modal-backdrop show" />
    </>
  )}
  ```
  - `modal show` — Bootstrap's classes that make the modal visible and
    positioned correctly
  - `d-block` — overrides Bootstrap's default `display: none` on `.modal`
  - `modal-backdrop show` — the dark overlay behind the modal

  The trade-off is losing the fade animation and automatic focus trapping. For
  a clinical tool used by staff who know the interface, this is acceptable.
  The full animation/accessibility version would use a library like
  `react-bootstrap` or `@headlessui/react`.

- **`window.confirm()` for delete confirmation (Step 5)**

  The Delete button uses `window.confirm()` — a browser-native dialog — rather
  than a custom confirmation modal. This is a deliberate simplicity choice: a
  second custom modal just for confirmation adds state and JSX for a low-value
  interaction. The native dialog is synchronous, accessible by default, and
  requires zero additional code.

  The trade-off: `window.confirm()` cannot be styled and looks different across
  browsers and operating systems. For a clinical tool where the look-and-feel
  is secondary to function, this is acceptable for the current milestone.

- **Modal extracted to a shared component to avoid repetition (Step 6)**

  The Surgeon page modal and the two modals on the Operation Types page are
  structurally identical: a Bootstrap dialog shell, a form, Cancel and Save
  buttons, and a loading state on Save. Copying the HTML three times would mean
  any future change (button wording, focus behavior, keyboard handling) needs to
  be applied in three places.

  `src/components/Modal.jsx` was created to hold this structure once. It accepts
  four props:
  - `title` — the header text
  - `onClose` — called when Cancel or the × button is clicked
  - `onSubmit` — the form's `onSubmit` handler (parent owns the mutation)
  - `isPending` — disables the Save button while the mutation is in flight
  - `children` — the form body (labels, inputs, error alerts)

  The `Surgeons.jsx` modal was refactored to use `Modal` at the same time it
  was written for `OperationTypes.jsx`, so the shared component was established
  before the second consumer was written rather than after.

  This is a deliberate, minimal abstraction: it covers exactly the structure
  that repeats and nothing else. The parent still owns all state and mutation
  logic, so each usage is independently testable and readable.

- **`ReferenceTable` sub-component keeps Operation Types page DRY (Step 6)**

  The Operation Types page has two tables (one for operation types, one for
  steps) that follow the same pattern: a heading with an optional Add button,
  a table with an empty-state row, and admin-only Edit/Delete action cells.

  Rather than writing this twice in the same file, a `ReferenceTable`
  function component is defined at the top of `OperationTypes.jsx`. It accepts
  `label`, `columns`, `rows`, and action callbacks as props. Each row is
  passed as `{ id, label, cells, raw }`:
  - `id` — used as the React key and passed to `onDelete`
  - `label` — used in the delete confirmation message
  - `cells` — array of values rendered into `<td>` elements in order
  - `raw` — the original API object, passed to `onEdit` so the modal can
    pre-fill the form

  `ReferenceTable` is not exported (not in `src/components/`) because it is
  specific to this page's layout. If a third table needed the same treatment,
  it would be worth promoting to a shared component.

- **Steps are global, not per-operation-type — layout decision (Step 6)**

  The original milestone plan described "a nested panel per operation type
  showing all globally defined steps." Once the data model was re-examined,
  this layout did not make sense: steps are not associated with operation types
  in the schema. Showing the full step list under every operation type would
  display the same list N times, which is confusing.

  The final layout uses two separate sections on the same page with a
  clarifying note: "Steps are globally defined procedure milestones shared
  across all operation types. The system suggests relevant steps based on what
  has been used historically for each operation type." This is honest about the
  data model and gives admins a single place to manage both resources.

- **Serializer field tests must be updated when new read-only fields are added (Step 7)**

  `test_serializers.py` had two tests that asserted an exact set of keys on
  each serializer: `assert set(data.keys()) == { 'id', 'operation_type', ... }`.
  Adding `operation_type_name` and `surgeon_name` broke both tests because the
  actual key set now has extra members.

  The fix was to add the new field names to the expected sets. The tests are
  correct to be this specific — they document the public contract of the
  serializer. If a field is accidentally added or removed in a future change,
  the test will catch it. The downside is that adding intentional new fields
  always requires a test update, which is a small maintenance cost worth paying
  for the contract clarity.

- **`operation_type_name` uses `source=` instead of `SerializerMethodField` (Step 7)**

  For read-only fields that are just an attribute of a related object, DRF's
  `CharField(source='related.attribute', read_only=True)` is more concise than
  a full `SerializerMethodField`:

  ```python
  # SerializerMethodField (more verbose, needed for null-safety logic)
  username = serializers.SerializerMethodField()
  def get_username(self, obj):
      return obj.user.username if obj.user else None

  # source= shorthand (sufficient when the relation is never null)
  operation_type_name = serializers.CharField(
      source='operation_type.operation_type', read_only=True
  )
  ```

  `operation_type` and `surgeon` are non-nullable foreign keys on
  `OperationInstance` (the model enforces this), so there is no null-access
  risk. `SurgeonSerializer` uses `SerializerMethodField` for `username` because
  `Surgeon.user` can be `None`.

- **Two separate query keys vs one query with client-side filtering (Step 7)**

  The Dashboard fetches active and completed operations as two separate
  `useQuery` calls with different query keys
  (`['operation-instances', 'active']` and `['operation-instances', 'completed']`).

  An alternative would be to fetch all operations once and split them
  client-side with `.filter(op => !op.complete)`. The trade-offs:

  **Single fetch, client-side split:**
  - One network request
  - React Query caches the combined result
  - After a mutation (e.g. completing an operation), the entire list must be
    invalidated and refetched — which is a larger payload

  **Two fetches, server-side filter:**
  - Two network requests on first load (both fire in parallel — React Query
    deduplicates concurrent fetches with the same key)
  - Each list is cached and invalidated independently
  - When an operation is completed (OCS2 → "Complete"), only the relevant
    cache entries need invalidation
  - The total payload is smaller: active operations are typically just a few
    records; completed operations are paginated to 25

  Timer 2.0 uses two fetches because the lists have meaningfully different
  sizes and update lifecycles, and the parallel network requests are
  imperceptible on a local network.

- **`completedData.count` vs `completed.length` for the section heading (Step 7)**

  The API response for a paginated list looks like:
  ```json
  { "count": 47, "next": "...", "previous": null, "results": [...25 items...] }
  ```

  `results.length` is at most 25 (the page size). `count` is the total number
  of records matching the filter — what we actually want to show in "Completed
  (47)". Using `results.length` would misleadingly show "Completed (25)" even
  when there are hundreds of past operations.

- **`StepInstanceSerializer` was missing `operation_instance` because it was only ever used read-only (Step 9)**

  The original serializer was designed to be nested inside
  `OperationInstanceDetailSerializer`:
  ```python
  steps = StepInstanceSerializer(many=True, read_only=True)
  ```
  In that context, `operation_instance` is implicit (you already know it because
  you're inside the parent), so it was not included in `fields`.

  OCS1 needs to `POST /api/v1/step-instances/` to create new step instances
  for the current operation. DRF's `create` action only writes fields that are
  declared in `fields`. Without `operation_instance` in the list, the POST body
  would silently ignore it and creation would fail with a NOT NULL constraint
  error from the database.

  Adding `'operation_instance'` to `fields` fixes both the standalone create
  AND the nested read (it now appears in the nested steps list too, which is
  redundant but harmless — the consumer can ignore it).

- **`useEffect` is required for navigation side effects triggered by data (Step 9)**

  Calling `navigate()` directly in the render body is invalid in React:
  ```jsx
  // Wrong — navigate is a side effect, not a render output
  if (operation?.complete) {
    navigate('/operations/:id/stats', { replace: true })
    return null
  }
  ```
  React will usually catch this, but in strict mode it fires renders twice and
  the behavior is undefined. The correct pattern is:
  ```jsx
  useEffect(() => {
    if (operation?.complete) navigate(`/operations/${id}/stats`, { replace: true })
  }, [operation?.complete, navigate, id])
  ```
  The `useEffect` runs after the render is committed to the DOM, so the
  navigation happens as a clean side effect. The `return null` guard after the
  hooks block is still needed so the rest of the component doesn't try to render
  with incomplete data while the redirect is in flight.

- **`addAllMutation` uses sequential `await` inside `mutationFn` to preserve order (Step 9)**

  When adding all suggested steps, order matters — step 1 must have `order=0`,
  step 2 `order=1`, and so on. If the requests were fired in parallel
  (`Promise.all`), the server would process them in an arbitrary order and the
  `order` values would be correct in the request bodies but the database writes
  might commit out of sequence (due to network jitter).

  Using `for...of` with `await` inside `mutationFn` ensures each step is
  created before the next one starts:
  ```js
  mutationFn: async steps => {
    for (const [i, step] of steps.entries()) {
      await client.post('/step-instances/', { ..., order: i })
    }
  }
  ```
  `onSuccess` fires once after the entire loop completes, triggering a single
  `invalidateQueries` that refetches the step list. This gives one clean
  re-render showing all suggested steps, instead of N re-renders (one per step
  added).

- **`page_size=100` on the steps dropdown query (Step 9)**

  DRF's `StandardResultsSetPagination` returns 25 records per page by default.
  An ASC could have more than 25 distinct step types, and a dropdown that only
  shows 25 of them would silently omit the rest.

  The backend sets `max_page_size = 100`, so `?page_size=100` is the maximum
  safe request. An ASC with more than 100 step types is extremely unusual; if
  that ever becomes a real constraint, the right fix is a server-side search
  input rather than a larger page size. For the MVP, 100 is sufficient.

- **The running clock drives live step elapsed for free (Step 10)**

  OCS2 needs two "ticking" values: the total time-in-room clock and the live
  elapsed for whichever step is currently active. Naively these might seem to
  need two timers. In React they need zero extra timers — one is enough.

  The `elapsed` state variable is updated by a single `setInterval` every
  second. Each update calls `setElapsed(...)`, which schedules a re-render.
  On each re-render, every table cell is recalculated — including the elapsed
  column, which calls `stepElapsed(si)` → `currentSeconds()`. Since
  `currentSeconds()` reads `new Date()` at call time, the step elapsed is
  always up-to-date at render time. No second timer needed.

- **`elapsed_time` on `StepInstance` is null until `complete_operation` (Step 10)**

  It might be tempting to display `si.elapsed_time` for the elapsed column.
  But that field is only computed and saved by `complete_operation()` in
  `services.py` — it runs at the end of the whole operation, not when each
  step's `end_time` is set.

  While the operation is in progress, a step can have both `start_time` and
  `end_time` set but `elapsed_time` still null. The `stepElapsed(si)` helper
  handles this by computing `end_time - start_time` locally when `elapsed_time`
  is null. This ensures the elapsed column is always populated once both times
  are known, regardless of whether the operation has been completed.

  The priority is: `si.elapsed_time` (API-computed, authoritative) → local
  compute from `end_time - start_time` → live `currentSeconds() - start_time`
  → `null` (displayed as `—`).

- **"Complete Operation" uses a `title` tooltip to explain why it's disabled (Step 10)**

  The "Complete Operation" button is disabled until every step has an `end_time`.
  For a nurse who doesn't know why the button is greyed out, a `title` attribute
  shows a tooltip on hover:
  ```jsx
  title={!allDone ? 'All steps must have an end time before completing' : undefined}
  ```
  Setting `title` to `undefined` (not an empty string) removes the attribute
  entirely when the button is enabled, so no tooltip appears in the enabled state.

- **`queryClient.setQueryData` eliminates the stats page loading flash (Step 11)**

  The `POST /operation-instances/:id/complete/` endpoint returns the full
  updated detail serializer (with nested steps and `dist_from_average`). Rather
  than letting the stats page refetch this same data, OCS2's `completeMutation`
  seeds the cache directly:
  ```js
  onSuccess: res => {
    queryClient.setQueryData(['operation-instance', id], res.data)
    queryClient.invalidateQueries({ queryKey: ['operation-instances'] })
    navigate(`/operations/${id}/stats`)
  }
  ```
  `setQueryData` writes data directly to the cache with the same key that
  `PostOpStats` will query (`['operation-instance', id]`). When `PostOpStats`
  mounts, React Query finds fresh data and skips the network request entirely.
  Without this, the stats page would briefly render with stale data
  (`complete: false`) while the refetch was in flight, showing a false warning.

- **`<a href>` instead of Axios for the CSV download (Step 11)**

  The CSV export is triggered via a plain HTML anchor tag rather than an Axios
  request:
  ```jsx
  <a href={`/api/v1/operation-instances/${id}/export-csv/`} className="btn btn-outline-secondary">
    Download CSV
  </a>
  ```
  There are two reasons:

  **Browser handles the download automatically.** When the server responds with
  `Content-Disposition: attachment; filename="..."`, the browser saves the
  file instead of navigating. Axios would need the response to be treated as a
  blob, then a synthetic `<a>` created and clicked, which is pointless extra
  code.

  **Cookies are sent automatically.** httpOnly cookies are attached by the
  browser to every request matching the cookie's domain and path — including
  anchor tag navigations. The `CookieJWTAuthentication` backend class will
  authenticate the request from the cookie. No special Axios configuration or
  token headers are needed.

  In both dev (Vite proxy) and production (nginx), the `/api/` prefix routes
  to the backend, so the URL resolves correctly in both environments.

- **Row color coding uses `Math.abs` so both directions read correctly (Step 11)**

  `dist_from_average` is positive when the surgeon is slower than average and
  negative when faster. The color coding uses the absolute value for the
  threshold checks:
  ```js
  const abs = Math.abs(dist)
  if (abs < 10) return 'table-success'
  if (abs < 25) return 'table-warning'
  return 'table-danger'
  ```
  A surgeon who is consistently faster (negative dist) is neither better nor
  worse than one who is slower by the same magnitude — the timer is measuring
  deviation, not performance. Green means "close to historical average" in
  either direction.

- **Multi-stage Dockerfile: the `npm ci` layer is cached unless dependencies change (Step 12)**

  The build stage copies `package*.json` first, runs `npm ci`, then copies the
  rest of the source:
  ```dockerfile
  COPY package*.json ./
  RUN npm ci
  COPY . .
  RUN npm run build
  ```
  Docker caches each `RUN` layer keyed on its inputs. By copying only the
  manifests before `npm ci`, the `node_modules` install only re-runs when
  `package.json` or `package-lock.json` changes. Changing any `.jsx` file (the
  common case during development) jumps straight to `npm run build` with
  `node_modules` already warm — the build is fast because `npm ci` is cached.

  If the `COPY . .` came first, any source file change would invalidate the
  `npm ci` layer and trigger a full dependency install on every build.

  `npm ci` is used instead of `npm install` because it reads only
  `package-lock.json` (no version resolution, no lockfile updates) and fails if
  the lockfile is out of sync. This makes container builds reproducible — the
  same package versions every time, regardless of when the image is built.

- **nginx location block ordering: specific paths before `/` (Step 12)**

  nginx matches `location` blocks from most-specific to least-specific. The
  `/api/` and `/admin/` proxy blocks are matched before the `location /` catch-
  all, so API requests are proxied to Django and static assets are served by
  nginx — the `try_files` fallback only applies to paths that didn't match a
  more specific block.

  Without the `try_files $uri $uri/ /index.html` fallback, a user who loads
  `/operations/123/ocs1` directly (or hits refresh on an authenticated page)
  would get nginx's default 404 because there is no file at that path on disk.
  With the fallback, nginx serves `index.html` and React Router handles the
  route client-side.

- **Backend `ports: 8000:8000` removed from docker-compose.yml (Step 12)**

  The backend service no longer exposes port 8000 on the host. All browser
  traffic reaches the backend through nginx on port 80. The backend is still
  reachable at `http://backend:8000` from other containers in the Docker
  Compose network (where `backend` is the service hostname).

  If you need to hit the Django API directly from the host (e.g. for `curl`
  debugging or to use the Django admin outside the nginx stack), temporarily
  add `ports: ["8000:8000"]` back to the backend service. Alternatively,
  `http://localhost/api/v1/` through nginx works for most purposes.

- **`MeView` needed `surgeon_id` so the form could pre-fill itself (Step 8)**

  The Begin Operation form needs to submit a `surgeon` ID. For non-admin users
  this should be their own surgeon record — no dropdown needed, the value is
  just posted silently.

  The only way the frontend knows the current user's surgeon ID without a
  second API call is if `/auth/me/` returns it. Adding `surgeon_id` to
  `MeView` means `AuthContext` stores it automatically (since `user` is just
  the full `/me/` response), making `user.surgeon_id` available everywhere
  without any extra fetching.

  The reverse OneToOne accessor (`request.user.surgeon`) raises
  `RelatedObjectDoesNotExist` (a subclass of `ObjectDoesNotExist`) when no
  Surgeon is linked to the User. This is Django's standard behavior for a
  OneToOne reverse relation that doesn't exist yet — unlike a nullable
  ForeignKey, which returns `None`. The `try/except Exception` in `MeView`
  catches this and returns `surgeon_id: null` instead of a 500 error.

- **`useState` initial value captures `user` at mount time (Step 8)**

  `BeginOperation` initialises form state with:
  ```js
  const [form, setForm] = useState({
    surgeon: isAdmin ? '' : String(user?.surgeon_id ?? ''),
    ...
  })
  ```
  `useState`'s initializer function runs exactly once — on the component's
  first render. Because `PrivateRoute` guarantees `user` is set before any
  authenticated page renders (it shows a loading spinner until `isLoading` is
  false), `user.surgeon_id` is available and stable at that first render. If
  `user` changed after mount (it won't in normal usage), the form would not
  update — but this is the correct behavior for a form that's already been
  opened.

- **`enabled: isAdmin` prevents unnecessary API calls for non-admin users (Step 8)**

  React Query's `enabled` option lets you skip a query conditionally:
  ```js
  useQuery({
    queryKey: ['surgeons'],
    queryFn: () => client.get('/surgeons/').then(r => r.data.results),
    enabled: isAdmin,
  })
  ```
  When `enabled` is `false`, the query is never executed — no network request,
  no loading state, `data` stays `undefined`. This matters because the surgeons
  list is only needed for the admin dropdown. Fetching it for every user who
  visits the Begin Operation page would be wasteful, especially since non-admin
  users never see the dropdown.

- **`<input type="hidden">` submits surgeon ID invisibly for non-admin users (Step 8)**

  The API requires a `surgeon` field on every `POST /api/v1/operation-instances/`.
  For non-admin users, this value is known (their own `surgeon_id`) and
  fixed — they should not be able to change it. Rather than including a visible
  but disabled dropdown (which would be confusing), the value is submitted via
  a hidden input:
  ```jsx
  <input type="hidden" name="surgeon" value={form.surgeon} />
  ```
  The `form` state contains the correct value; the `setField` handler (which
  uses `e.target.name`) updates it just like any other field if needed. The
  hidden input keeps the form's submit logic uniform — `handleSubmit` always
  just submits `form` as-is regardless of which user type is logged in.

- **`CookieJWTAuthentication` silently swallows cookie validation errors (Step 2)**

  In the `authenticate()` method, the cookie branch wraps validation in a
  bare `except Exception: return None`. This is intentional.

  DRF's authentication contract is: return `(user, token)` if authenticated,
  return `None` if this authenticator does not apply, raise
  `AuthenticationFailed` if the credentials are present but invalid. The cookie
  branch uses `return None` on failure rather than raising `AuthenticationFailed`
  because an invalid or expired cookie should silently fall through to
  "unauthenticated" — it will then trigger a 401 from the permission layer,
  which the Axios interceptor catches and converts into a silent token-refresh
  attempt. Raising `AuthenticationFailed` would short-circuit that flow and
  immediately return a 401 to the browser, bypassing the refresh attempt.
